# Mandatory standard format — Docmail

**Every docmail MUST contain these 6 sections in this order:**

## 1. Header
```html
<h1>[Subject]</h1>
<div class="subtitle">[Generating skill] &bull; [Project] &bull; [Readable date]</div>
```
- Generating skill = the skill/command that produced the content (e.g. `/docmail`, `/docmail arbitrage`)
- Project = the project concerned (detect from cwd or context)

## 2. Timeless recap
```html
<div class="card card-context">
  <h3>Context</h3>
  <p>2-3 lines about the project, understandable in 3 months by someone without the context.</p>
</div>
```
This recap does not change from one docmail to the next for the same project. Read your project context notes if available.

## 3. Triptych
```html
<div class="card card-accent">
  <h3>Problem</h3><p>1-2 lines</p>
  <h3>Reasoning</h3><p>1-3 simplified lines</p>
  <h3>Solutions</h3><p>In progress or proposed, 1-3 lines</p>
</div>
```

## 4. Timestamp
```html
<div class="section-note">
  Monday, April 3rd &mdash; session started at 2pm, docmail generated at 2:40pm
</div>
```
- Day of week spelled out, full date
- Session start time (approximate from context)
- Docmail generation time

## 5. Content
The body of the docmail — specific to each invocation. Use the design system components (`design-system.md`) according to the content type.

## 6. Technical signature
```html
<footer>
  Generated by Claude Code &bull; [Model] &bull; [Generation date]
  <br>Project: [name] &bull; Branch: [git branch if available]
</footer>
```
Do NOT include token counts or session duration: unreliable from context, risk of invention. Model = exact ID. Branch = output of `git rev-parse --abbrev-ref HEAD` if in a repo, otherwise omit.
